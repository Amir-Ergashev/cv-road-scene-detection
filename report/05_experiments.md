# 3.6 Эксперименты

## 3.6.1 Условия проведения экспериментов

Все эксперименты проводились в едином окружении для обеспечения
сопоставимости результатов:

- **Оборудование:** NVIDIA RTX 4070 Laptop GPU (8 ГБ видеопамяти),
  CPU Intel Core i7-14700HX, 16 ГБ RAM.
- **Программное обеспечение:** Python 3.13, PyTorch 2.6.0 (CUDA 12.4),
  torchvision 0.27.1, ultralytics 8.4.79, effdet, transformers 5.12.1.
- **Данные:** единое разбиение train/val/test (2080/445/447 изображений,
  раздел 3.4), фиксированное для всех моделей (`seed=42`).

## 3.6.2 Воспроизводимость

Для всех моделей зафиксирован `seed=42` на этапе разбиения датасета
(train/val/test split, раздел 3.4) и при инициализации обучения
(где это поддерживается API библиотеки — YOLOv8/ultralytics).
Конфигурация гиперпараметров для каждой модели зафиксирована в
соответствующих файлах `train_<model>.py` репозитория проекта.

## 3.6.3 Особенности подбора гиперпараметров по моделям

В процессе обучения для нескольких моделей потребовалась корректировка
изначально выбранных гиперпараметров — это типичная практика при работе
с новыми архитектурами на собственном датасете, и сами найденные проблемы
представляют отдельный практический результат:

**SSD300.** Изначальный learning rate 0.002 в сочетании с mixed precision
(AMP) приводил к численной нестабильности обучения (loss становился NaN
на всех эпохах). Решение: отключение AMP для данной модели, снижение
learning rate до 0.0015, добавление linear warmup на первые 200 шагов и
gradient clipping (max_norm=5.0). После этого обучение стабилизировалось
(loss снизился с ~46 до ~5.85 на протяжении 15 эпох).

Дополнительно было скорректировано пороговое значение уверенности
предсказаний (`score_thresh`): дефолтное значение библиотеки torchvision
(0.01) приводило к сохранению избыточного количества слабых предсказаний
на изображение (до 200), что искусственно занижало Precision при оценке
метрик. Значение было увеличено до 0.2, что несколько улучшило
Precision, однако итоговая точность модели осталась существенно ниже
остальных архитектур — подробный анализ приведён в разделе 3.8
"Обсуждение".

**EfficientDet и DETR.** Для обеих моделей применён аналогичный приём
linear warmup (200 шагов для EfficientDet) для стабильности обучения
трансформерной/составной архитектуры на этапе разогрева весов
случайно инициализированной головы классификации.

**DETR.** Изначальный размер изображения (800/1333 px по умолчанию)
приводил к непропорционально долгому времени обучения (~40 минут на
эпоху) при ограниченных вычислительных ресурсах. Размер уменьшен до
480/640 px, число эпох — до 8 (вместо изначальных 15), что сократило
время обучения до ~115 секунд на эпоху без потери принципиальной
возможности оценить архитектуру.

## 3.6.4 Эксперимент с разными гиперпараметрами: YOLOv8 на CPU vs GPU

Дополнительно проведён сравнительный эксперимент влияния вычислительного
устройства на скорость обучения для модели YOLOv8n:

| Конфигурация | Устройство | Размер изображения | Эпохи | Время обучения |
|---|---|---|---|---|
| CPU-конфигурация | CPU (Intel Core i7-14700HX) | 416×416 | 20 | 53 минуты |
| GPU-конфигурация | GPU (RTX 4070) | 640×640 | 50 (early stop на 11) | 2.76 минуты |

Несмотря на больший размер изображения и потенциально большее число эпох,
GPU-конфигурация завершила обучение почти в 20 раз быстрее, что
подтверждает критическую роль GPU-акселерации при обучении моделей
компьютерного зрения, особенно при необходимости провести серию
экспериментов с несколькими архитектурами в ограниченные сроки.
